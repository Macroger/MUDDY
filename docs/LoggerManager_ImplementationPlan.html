What this phase finishes

The current branch set out to remove all non-strongly-typed event dispatch from the bus. Two things remain: EventEnvelope (a wrapper class that was used to tag events with metadata before publishing) still exists and is still referenced by PacketLogger and FileLogger. SubscribeAll was removed but needs to come back — correctly this time — as a separate _observers collection with no EventMessageType parameter, used exclusively by loggers.

This phase cleans those up so the branch is in a consistent state before committing: EventEnvelope gone, SubscribeAll back with the right signature, all loggers updated to use direct is-pattern matching on the raw event object.

Phase 2 progress 0 / 9 steps complete
What this phase builds

After Phase 1 the loggers are passive full-event observers — they see everything but have no intelligence about what to do with it. At scale (many players, high event volume) this is wasteful and noisy. This phase introduces a LoggerManager that owns all logger instances, controls their capture scope at runtime, and exposes its state to the GUI.

The key design decisions made in the design discussion: loggers become dumb writers — they receive an event and write it, nothing more. All decisions about what to capture live in the manager. The GUI sends a SetLoggerModeEvent on the bus; the manager responds by disposing the old subscription token and registering a new one on the affected logger. A LoggerModeChangedEvent is published back so the GUI can update its status display without polling.

Key design decisions

Why loggers are dumb writers

Moving capture-scope logic into loggers creates two responsibilities in one class. The manager is the single place to change what is logged — loggers never need to know about modes or bus subscriptions.

Why mode changes go through the bus

The GUI already communicates with all server components via the bus. Adding a direct reference from the GUI to the LoggerManager would break the decoupling pattern. The bus event is the correct channel.

Why SubscribeCategory is added now

It was deferred as speculative earlier, but LoggerMode.Category is a concrete use case that needs it. This is now the right time to add it.

Why tokens live in the manager

Loggers replacing their own tokens on mode changes is possible but creates a lifecycle coupling between the logger and the bus. The manager owns subscriptions so loggers have zero bus awareness.

    Files: Shared/Logging/FileLogger.cs, Shared/Logging/EventBusLogger.cs

    Remove bus subscriptions from loggers entirely. Loggers become simple write targets that the manager calls.

    • Expose a public Write(object evt) method (or keep HandleEvent as internal) that the manager calls
    • Keep IDisposable on FileLogger — it still needs to flush and close the StreamWriter
    • Keep the Name property (a string identifier) so the manager and GUI can refer to each logger by name
    ⚠ DI registrations will need updating — loggers no longer take IEventBus in their constructors.

  • 06 Create LoggerManager implementation Create ▼

    File: Shared/Logging/LoggerManager.cs (or Server.Core/Logging/ if it needs server-specific dependencies)

    The manager owns all logger instances and their current subscription tokens.

    • Constructor accepts IEventBus and a list of logger instances
    • Maintains a Dictionary<string, (ILogger logger, ISubscriptionToken? token, LoggerMode mode, EventMessageType? category)> keyed on logger name
    • On construction: subscribe to SetLoggerModeEvent on a Logging or System category; publish initial LoggerStatusSnapshot
    • SetMode: dispose existing token, create new subscription based on mode (SubscribeAll / SubscribeCategory / nothing), update state, publish LoggerModeChangedEvent
    • Implements IDisposable: disposes all tokens and all loggers
    ⚠ Add Logging to EventMessageType enum if a dedicated category for logger control messages is desired, to avoid routing them through System.
  • 07 Update Server.Core DI registration Register ▼

    Wire up the new types in the server's dependency injection composition root.

    • Register FileLogger and EventBusLogger as singletons (no longer need IEventBus passed in)
    • Register ILoggerManager → LoggerManager as a singleton, passing in the bus and the logger instances
    • Remove any old direct bus subscriptions that loggers previously self-registered in their constructors
    • Ensure LoggerManager is resolved early in startup so loggers are active before other services begin publishing
  • 08 Add Logger status panel to Server.GUI Create ▼

    Files: Server.GUI/MainWindow.xaml (or a new panel/page), Server.GUI/MainWindow.xaml.cs

    A GUI element that shows each logger's name, current mode, and a control to change the mode.

    • Subscribe to LoggerModeChangedEvent and LoggerStatusSnapshot with typed Subscribe<T> — no reflection, no SubscribeAll in the GUI
    • Dispatch all UI updates through DispatcherQueue.TryEnqueue matching the existing pattern in MainWindow.xaml.cs
    • For each logger row: display Name, Mode badge, Category (if applicable), and a dropdown or button group to select a new mode
    • On mode selection: publish SetLoggerModeEvent on the bus — GUI never calls the manager directly
    • Populate initial state from LoggerStatusSnapshot published by the manager at startup
    ⚠ Do NOT use reflection or dynamic property access on event objects in the GUI — this was the original crash cause. All display strings must be built from explicitly typed event properties.
