Why this approach was chosen

The MUDDY server uses a synchronous, in-process EventBus (BasicEventBus) where Publish<T> iterates over a HashSet<IEventSubscriber> and calls each handler sequentially on the same thread. This means handler execution time directly contributes to tick latency — a slow handler silently steals budget from every other system that fires in the same tick.

Because all dispatch is synchronous and single-threaded, a Stopwatch wrap around each handler.Invoke call gives 100% accurate wall-time attribution per handler, per event type with zero threading ambiguity. No external agent is needed for this signal.

An external profiler (Visual Studio CPU Usage / dotnet-trace) answers why a handler is slow (GC pressure, I/O blocking, hot inner loops), but it cannot fire a runtime alert when a budget is exceeded during live play. The two-layer design gives you both: the in-process monitor fires a bus event you can log or display in the Server GUI; the external profiler is reserved for root-cause diagnosis when the monitor flags a problem.

Two-layer design

Layer 1 — In-Process Monitor (build now)

  • Per-handler wall-time via Stopwatch
  • Tick budget violation alerts at runtime
  • Total Publish dispatch time
  • Handler call frequency tracking
  • Zero external dependencies
  • Cannot explain why a handler is slow
  • No CPU/thread contention visibility
  • No GC pause attribution

Layer 2 — External Profiler (on demand)

  • CPU hotspot drill-down (VS / dotnet-trace)
  • GC pressure and allocation analysis
  • Thread contention identification
  • I/O wait attribution
  • Tick scheduling jitter (pre-publish)
  • Cannot fire live runtime alerts
  • Not suitable for continuous production use

    Add a new Performance enum value so that performance events have their own routing channel on the bus, keeping them isolated from Domain, System, and other categories.

    • No other changes to this file
  • 02 Create PerformanceEvents.cs Create ▼

    File: Shared/EventBus/EventTypes/PerformanceEvents.cs

    New event container class following the same nested-sealed-record pattern used by NetworkEvents, ChatEvents, etc.

    • HandlerSlowEvent(string HandlerTypeName, string EventTypeName, TimeSpan Elapsed, TimeSpan Budget) — fired when a single handler exceeds its budget
    • PublishSlowEvent(string EventTypeName, int HandlerCount, TimeSpan TotalElapsed, TimeSpan Budget) — fired when the full Publish call for an event type exceeds the per-publish budget (tick-level signal)
  • 03 Create PerformanceMonitorOptions.cs Create ▼

    File: Shared/Performance/PerformanceMonitorOptions.cs

    A plain configuration record (or class) injected into the monitor at startup. Keeps all budget values in one place and out of the monitor logic.

    • bool IsEnabled — master on/off switch (default true)
    • TimeSpan DefaultHandlerBudget — per-handler threshold, e.g. 5 ms
    • TimeSpan DefaultPublishBudget — per-Publish threshold, e.g. 16 ms (one 60fps tick)
    • IReadOnlyDictionary<string, TimeSpan> PerEventBudgetOverrides — optional per-event-type overrides (keyed on event type name)
  • 04 Create IPerformanceMonitor.cs Create ▼

    File: Shared/Performance/IPerformanceMonitor.cs

    A thin interface so the event bus depends on an abstraction and the monitor can be swapped or mocked in tests.

    • void RecordHandlerTiming(string handlerTypeName, string eventTypeName, TimeSpan elapsed) — called by the bus once per handler invocation
    • void RecordPublishTiming(string eventTypeName, int handlerCount, TimeSpan elapsed) — called by the bus once per Publish call, after all handlers have run
  • 05 Create BasicPerformanceMonitor.cs Create ▼

    File: Shared/Performance/BasicPerformanceMonitor.cs

    Concrete implementation. Accepts IEventBus and PerformanceMonitorOptions via constructor. Compares recorded durations against configured budgets and publishes alert events when thresholds are exceeded.

    • Implement RecordHandlerTiming: if elapsed > budget, publish PerformanceEvents.HandlerSlowEvent on EventMessageType.Performance
    • Implement RecordPublishTiming: if elapsed > budget, publish PerformanceEvents.PublishSlowEvent on EventMessageType.Performance
    • Resolve per-event budget from PerEventBudgetOverrides first, fall back to DefaultHandlerBudget
    • Guard with IsEnabled check at the top of each method — zero overhead when disabled
    • No internal state accumulation needed for MVP; add counters/histograms later if required
  • 06 Instrument BasicEventBus.Publish with Stopwatch Modify ▼

    File: Shared/EventBus/BasicEventBus.cs

    Inject an optional IPerformanceMonitor? into BasicEventBus via constructor (defaults to null — existing DI registrations that don't pass a monitor continue to work without change).

    • Add private readonly IPerformanceMonitor? _monitor; field
    • Add optional constructor parameter: IPerformanceMonitor? monitor = null
    • Inside the handler loop, wrap handler.Invoke(newEvent) with Stopwatch.GetTimestamp() before/after (avoid allocating a Stopwatch instance per call — use the static timestamp API)
    • After each invoke: _monitor?.RecordHandlerTiming(handler.GetType().Name, eventType.Name, elapsed)
    • After the loop exits: _monitor?.RecordPublishTiming(eventType.Name, handlerCount, totalElapsed)
    • Timing calls must be outside the existing per-handler try/catch so exceptions don't corrupt elapsed measurements
  • 07 Register BasicPerformanceMonitor in Server.Core DI Register ▼

    File: Server.Core DI registration (wherever BasicEventBus is currently wired up as a singleton)

    • Register PerformanceMonitorOptions as a singleton with desired budget values
    • Register IPerformanceMonitor → BasicPerformanceMonitor as a singleton
    • Update the BasicEventBus registration to resolve and pass IPerformanceMonitor into its constructor
    • Optionally subscribe a lightweight logging handler to PerformanceEvents.HandlerSlowEvent on EventMessageType.Performance so violations appear in the server log and GUI event feed immediately
